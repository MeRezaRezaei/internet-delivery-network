Root Cause Analysis
Phantom Traffic: The UUIDs failing in your logs (318728b2..., 988662da..., etc.) do not exist anywhere in your configurations. They are residual traffic from old clients, forgotten devices, or scanners probing your server via WebSocket (which typically uses a UUID as the URL path).

Path Collision: Because your xhttp inbound is listening on the root "path": "/", it intercepts these old WebSocket probes. The xhttp protocol attempts to parse the URL path (the UUID string) as a numerical Session ID, triggering the strconv.ParseUint panic when it fails to convert the letters and hyphens into an integer.

Resolution
Change the path from / to a unique endpoint (e.g., /x-tunnel) on both your portal and bridge. This isolates your xhttp traffic and forces the server to drop the old /UUID probes, eliminating the errors.

Note on your bridge file: Tunnel 100 incorrectly places padding variables at the root of xhttpSettings instead of inside "extra". Additionally, all tunnels incorrectly nest "xmux" inside extra (standard Xray ignores this placement).

The Issue
systemctl disable and systemctl enable do not support wildcard/glob patterns (*). These commands directly manipulate filesystem symlinks and require exact, explicit unit names.

(Note: State commands like stop, start, and restart generally do accept wildcards if properly quoted, but management commands do not).

The Solution
To dynamically find and disable/stop all instantiated xray@ services, you need to query systemd for the active instances and pass them to the disable --now command.

Run this exact command:

Bash
systemctl list-unit-files "xray@*.service" --plain --no-legend | awk '{print $1}' | xargs -r systemctl disable --now
Breakdown of the Command
systemctl list-unit-files "xray@*.service" --plain --no-legend: Grabs a clean text list of every xray@ unit registered in systemd.

awk '{print $1}': Extracts just the unit names from the first column.

xargs -r systemctl disable --now: Feeds those exact names into your original command one by one, safely stopping and disabling all of them.

Here is exactly what was updated in the Python script to resolve the issues:

1. Fixed the PEM Base64 Padding (The Root Cause)

The embedded SSL certificate in the Python script had malformed Base64 line breaks. When a PEM string is broken into lines incorrectly (creating segments that are not proper multiples of 4), strict SSL parsers will crash with an InvalidData(InvalidPadding) error. The certificate array was realigned to restore proper Base64 formatting.

2. Corrected the XHTTP extra Schema

In the generate_portal_config function, variables like noPadding, xPaddingObfsMode, xPaddingBytes, and scMaxConcurrentPosts were improperly placed at the root of the xhttpSettings block. They were moved inside the extra object to comply with the required Xray protocol schema.

3. Linked the Portal Path Variable

The Portal generator was previously hardcoded to use "path": "/". It was updated to properly consume the dynamic path variable (args.path). This ensures that any custom path passed via the CLI (like --path "/x-tunnel") correctly applies to both the Bridge and the Portal simultaneously.

You are absolutely right. Those logs prove that the portal and the bridge were failing to handshake properly.

Here is exactly what those errors mean and how the latest script fixed them:

1. The invalid padding ... length:0 Error

The Problem: The bridge and the portal had completely mismatched padding configurations. The bridge was using standard extra-block padding ("xPaddingBytes": "500-1500"), while the portal was configured with a complex, header-based padding schema (xPaddingPlacement: "header", "xPaddingHeader": "X-Padding"). Because they weren't speaking the exact same padding "dialect," the portal couldn't read the padding size the bridge was sending (resulting in length: 0) and dropped the connection.

The Fix: I completely stripped the complex header-padding rules from the portal's configuration. In the new script, the portal's xhttpSettings is an exact, 1-to-1 mirror of the bridge's xhttpSettings. They now share the exact same extra block, ensuring they negotiate padding perfectly.

2. The ParseUint ... invalid syntax Error

The Problem: In XHTTP mode, the protocol uses the URL path to transmit numerical session IDs. The log shows it tried to read "8c2a5222-da62-4b35-8a94-98924a3a3585" (a UUID) but crashed because it expects an integer (ParseUint), not letters and hyphens. This happens because legacy clients (like standard WebSockets) encode their client UUID directly into the URL path. Since your path is set to /, your server intercepts these old WebSocket requests, tries to parse the UUID as an XHTTP session ID, and fails.

The Fix: While this error looks scary, it is actually harmless to your actual tunnels. Now that the padding error is fixed, your genuine bridge traffic will connect successfully. Those ParseUint errors are just the portal forcefully rejecting legacy/rogue traffic that knocks on the door incorrectly. (Though, as mentioned before, changing your path to something unique like /x-tunnel is the only way to make those specific log warnings disappear completely).

Explanation of the Fix
Marzban acts as a strict middleman validator. While core Xray is perfectly happy accepting raw certificates as JSON arrays, Marzban uses Python's cryptography library internally to validate the configuration before saving it to the database.

This library is incredibly strict. If it parses a raw Base64 certificate array from a JSON file and even a single line is one character off from the standard 64-character limit (which happens easily when embedding strings in JSON), it instantly triggers the InvalidData(InvalidPadding) panic and refuses to save.

The Solution:
I have completely removed the raw certificate block from the Python generator and reverted it back to your original, clean format:

JSON
"certificates": [
  {
    "certificateFile": "/opt/node/certs/ssl_bundle.pem",
    "keyFile": "/opt/node/certs/ssl_bundle.pem"
  }
]
This forces Marzban to simply look at the file path on your server rather than trying to parse a massive chunk of raw Base64 text in the JSON. This completely bypasses the validator bug and will allow you to save the config immediately.

This version generates fresh, self-signed RSA certificates programmatically
to ensure Marzban's cryptography validator accepts the configuration.

import json

from cryptography import x509
from cryptography.hazmat.primitives import hashes, serialization
from cryptography.hazmat.primitives.asymmetric import rsa

import datetime

def generate_new_cert():
    """Generates a fresh self-signed cert and returns PEM strings."""
    key = rsa.generate_private_key(public_exponent=65537, key_size=2048)
    subject = issuer = x509.Name([x509.NameAttribute(NameOID.COMMON_NAME, "i-01.doctel.ir")])
    cert = x509.CertificateBuilder().subject_name(subject).issuer_name(issuer).public_key(key.public_key()).serial_number(x509.random_serial_number()).not_valid_before(datetime.datetime.utcnow()).not_valid_after(datetime.datetime.utcnow() + datetime.timedelta(days=3650)).add_extension(x509.SubjectAlternativeName([x509.DNSName("i-01.doctel.ir")]), critical=False).sign(key, hashes.SHA256())
    
    key_pem = key.private_bytes(encoding=serialization.Encoding.PEM, format=serialization.PrivateFormat.TraditionalOpenSSL, encryption_algorithm=serialization.NoEncryption()).decode('utf-8')
    cert_pem = cert.public_bytes(serialization.Encoding.PEM).decode('utf-8')
    
    # Format for JSON Xray compatibility (stripping headers/footers)
    def clean_pem(pem):
        return [line for line in pem.split('\n') if line and not line.startswith('-----')]
        
    return clean_pem(cert_pem), clean_pem(key_pem)

def generate_new_tls_cert(domain):
    """Generates a valid X.509 certificate and returns it in the format Xray expects."""
    key = rsa.generate_private_key(public_exponent=65537, key_size=2048)
    subject = issuer = x509.Name([x509.NameAttribute(NameOID.COMMON_NAME, domain)])
    cert = x509.CertificateBuilder().subject_name(subject).issuer_name(issuer).public_key(key.public_key()).serial_number(x509.random_serial_number()).not_valid_before(datetime.datetime.utcnow()).not_valid_after(datetime.datetime.utcnow() + datetime.timedelta(days=3650)).add_extension(x509.SubjectAlternativeName([x509.DNSName(domain)]), critical=False).sign(key, hashes.SHA256())
    
    key_pem = key.private_bytes(encoding=serialization.Encoding.PEM, format=serialization.PrivateFormat.TraditionalOpenSSL, encryption_algorithm=serialization.NoEncryption()).decode('utf-8')
    cert_pem = cert.public_bytes(serialization.Encoding.PEM).decode('utf-8')
    
    # Return as properly formatted arrays
    return [line for line in cert_pem.split('\n') if line and not line.startswith('---')], \
           [line for line in key_pem.split('\n') if line and not line.startswith('---')]
